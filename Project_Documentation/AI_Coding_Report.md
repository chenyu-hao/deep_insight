# AI 编程使用报告（用于导出 PDF）

## 1. 使用 AI 编程的目标
在毕设周期内，使用 AI 编程助手提升开发效率与代码质量，重点用于：
- 快速定位前后端联调问题；
- 生成可维护的重构方案（SWR 缓存、对齐聚类字段统一等）；
- 优化 UI/交互体验（加载动画、布局稳定性、可解释性展示）。

## 2. 典型问题与解决案例（节选）

### 案例 A：热点平台筛选“后端有数据、前端显示暂无数据”
**问题**：HN/贴吧采集成功，但前端筛选为 0。  
**定位过程**：
- 增加后端统计日志：raw_counts / cluster_counts，确认 HN/贴吧进入对齐聚类；
- 发现前端只拿 top-N 结果且本地过滤依赖 top-N evidence，导致长尾平台被截断；
**解决**：
- 在对齐结果增加 `platform_ids`（聚类包含平台全集，不受 top-N 截断影响）；
- 前端筛选优先使用 `platform_ids`；
- 调整接口返回数量，避免 HN/贴吧排在 200 名后导致筛空。

### 案例 B：热榜刷新等待时间长、用户体验卡顿
**问题**：刷新时需要等待全量采集与对齐，阻塞 UI。  
**解决**：
- 采用 SWR（stale-while-revalidate）：优先返回缓存数据，后台异步刷新；
- 前端增加“refreshing 轮询”策略，在后台刷新完成后自动获取新数据。

### 案例 C：主页 AI 图片加载导致布局抖动（CLS）
**问题**：图片未加载时容器高度塌陷，文字上移。  
**解决**：
- 固定容器比例占位（aspect-ratio）；
- 图片绝对定位铺满（object-cover）；
- skeleton/加载遮罩 + `loading="lazy"`、`decoding="async"`。

### 案例 D：解释性展示不足，无法体现 Agent 贡献
**问题**：评委无法直观看到“这是 Agent/LLM 做的”。  
**解决**：
- 后端在生成式数据接口返回 `agent_name/used_llm/cache_hit/llm_reasoning`；
- 前端增加 Agent 信息卡片与 LLM 内容片段展示；
- 统一把“过程信息”放到合理位置、可折叠展示。

## 3. 使用方式与规范
- **先定位再修改**：先用日志/字段校验定位数据链路，再做最小可行修复。
- **可回滚设计**：通过配置项（如安全模式、缓存开关）控制策略，不暴露给前端用户。
- **交付友好**：清理多余测试脚本、补齐 README 与文档目录，便于评审复现。

## 4. 效果总结
- 提升联调效率：缩短“定位→修复→验证”闭环时间；
- 提升可演示性：增加可解释字段与链路展示，便于答辩；
- 提升工程质量：修复隐性 bug（数组原地 sort、定时器泄漏、缓存接口字段不一致等）。
